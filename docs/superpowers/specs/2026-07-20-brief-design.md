# The Brief — Design Spec

**Date:** 2026-07-20
**Status:** Approved design, pending implementation plan
**Milestone:** 3 of 6 (chief-of-staff roadmap)

## Product Vision (context)

Velo is being reshaped around the chief-of-staff model: email is state, not
a stream. The client maintains a model of the user's world — conversations,
obligations, records, noise — and renders *that*; messages are evidence you
drill into.

The Brief is the keystone surface: opening Velo shows a written morning-memo
that has already processed everything. On a good day the user reads four
sentences, feels done, and closes the app.

Roadmap: 1. Heavy prune (done) · 2. Noise triage (Home/Focus/Feed shipped;
Claude classification + per-sender overrides deferred) · **3. The Brief
(this spec)** · 4. Waiting-on ledger · 5. Records vault · 6. The Desk.

Decisions locked during brainstorming (2026-07-20):

- **The Brief IS the home.** The Focus/Feed tabs Home moves one level down,
  intact, as the raw fallback view.
- **Auto-regenerate after sync**, debounced, only when Focus-relevant mail
  changed. Opening the app never waits on a model call — the cached memo
  renders instantly.
- **Narrative memo + receipts.** Chief-of-staff prose (what needs you, what
  happened, notable feed items, a "coming up" line for spotted dates),
  every claim linked to its source thread, ending with a deterministic
  "filed" footer.
- **Full focus bodies, capped**, are sent to the model; feed items send
  subject + sender only.
- **Architecture B: extract, then compose.** Per-thread cached extractions;
  a small compose call renders the memo from them.
- **Provider-agnostic.** The Brief must work with any provider behind the
  existing `services/ai` abstraction — including local models via Ollama.
  Claude with the user's API key is the launch configuration, not a
  dependency.

## Goal

Ship Brief v1: a per-account narrative memo as the app's landing surface,
generated by a two-stage extract/compose pipeline over Focus mail, cached,
regenerating automatically after relevant syncs, with every claim linked and
verified, degrading gracefully when no AI provider is configured.

## UX

### BriefPage (new home)

- Route `/mail/brief` becomes the default (`/` redirects there; `g h` goes
  here). The current tabbed Home stays at `/mail/home`; `g i` still reaches
  the classic Inbox view. Sidebar: "Brief" replaces "Home" at the top;
  "Home" (tabs) moves to second position.
- Renders inside `MailLayout` in place of the list pane (same contract as
  `HomePage` — reading pane still works when a linked thread is opened).
- Content: dateline ("Sunday, July 20"), then the memo. Claims render as
  accent-colored links; clicking navigates to the thread (`navigateToThread`).
- Footer: **Filed** — deterministic receipts from triage, no model
  involvement: "12 to the feed (3 calendar · 2 FYI · 7 likely junk)",
  linking to the corresponding feed tabs.
- Staleness chip in the corner: "as of 9:41" plus a subtle spinner while a
  regeneration is in flight; the old memo stays visible until the new one
  lands (no flash-of-empty).
- Manual refresh: `F5` on this view (and a refresh affordance on the
  staleness chip) forces a regeneration — it bypasses the change-gating but
  still reuses valid cached extractions, so it costs one compose call plus
  any genuinely stale extractions.

### Fallbacks (all deterministic)

| Condition | Behavior |
|---|---|
| No AI provider configured / no API key | BriefPage renders a one-time setup card (link to Settings > AI) above the tabbed Home content, which becomes the effective landing view |
| Provider call fails | Keep last good memo with an "as of" timestamp and a quiet retry-on-next-sync; error detail behind the staleness chip |
| No focus mail at all | Deterministic memo: "Nothing needs you." + filed footer (no model call) |
| First run (no cache) | Skeleton while the first generation runs; if it takes >10s, show the tabbed Home content with a "brief is being written" note |

## Architecture

Two stages, both through `getActiveProvider()` from
`services/ai/providerManager`. **No provider-specific API anywhere in the
brief pipeline** — plain `AiCompletionRequest` completions only, no tool
use, no streaming, so any provider (including Ollama-served local models)
works. JSON parsing must be defensive (fenced-block stripping, single
retry with a "return only JSON" nudge) because local models are sloppier
than Claude.

### Stage 1 — per-thread extraction (cached)

For each Focus-classified thread in the current window (unread OR last
message within 48h, cap 30 threads):

- **Input:** thread subject, participants, and per-message bodies (each
  message truncated to ~2,000 chars, thread capped at ~8,000 chars, newest
  messages win).
- **Output (JSON, schema versioned):**
  ```json
  {
    "summary": "1-2 sentences of what happened, newest development first",
    "needsYou": true,
    "why": "she asked you to pick a date",
    "dates": [{"iso": "2026-07-25", "what": "registration renewal deadline"}],
    "people": ["Sarah Chen"]
  }
  ```
- **Cache:** `ai_cache` row `(accountId, threadId, "brief_extract_v1")`.
  Content stores `{stateKey, extraction}` where `stateKey =
  lastMessageAt + ":" + messageCount`; a thread is re-extracted only when
  its stateKey changes. A sync touching two threads costs two extraction
  calls.

### Stage 2 — compose (small, every regeneration)

- **Input:** the extraction manifest (threadId → extraction) for the
  window, feed metadata (subject + sender + category for today's feed
  arrivals, cap 20), and today's date. No bodies.
- **Prompt contract:** write the memo in plain confident prose, ≤ ~180
  words; reference threads **only** via `[text](thread:THREAD_ID)` tokens
  using IDs from the manifest; needs-you items first, then developments,
  then at most one sentence on notable feed items, then at most one
  "coming up" sentence built from extracted dates. Never invent facts not
  present in extractions.
- **Link validation (deterministic):** after generation, every
  `thread:ID` token is checked against the manifest; unknown IDs render as
  plain text and are counted; if >20% of links are invalid, the memo is
  rejected and the last good one is kept. This makes link trust a
  validation problem, not a prompting hope.
- **Cache:** composed memo stored in `ai_cache` as
  `(accountId, "__brief__", "brief_memo_v1")` with generation timestamp
  and the manifest hash it was built from.

### Trigger pipeline (`services/brief/briefManager.ts`)

- Listens to `velo-sync-done` (same debounce pattern as the views, 2s).
- Computes the focus window; regenerates only if (a) any window thread's
  stateKey differs from its cached extraction, or (b) the manifest set
  hash differs from the one the cached memo was built from. Feed-only
  syncs are free.
- Serialized: one generation in flight per account; a newer trigger during
  flight marks dirty and reruns once.
- Emits `velo-brief-updated` for BriefPage to re-read the cache.

### Cost profile

Steady state per relevant sync: N changed-thread extractions (usually 0-2)
+ 1 compose call over ~30 short JSON blobs. Model: whatever Settings > AI
has active (existing provider/model pickers; Claude Haiku default). No new
settings in v1 beyond what exists.

## What This Spec Does NOT Change

- Triage classifier, Focus/Feed tabs, archive-all — untouched; tabs remain
  the raw fallback view.
- Notifications/badge — already focus-only; the Brief adds no notification
  surface in v1 (morning-notification delivery is a later idea).
- No new DB migrations: `ai_cache` and `settings` cover all storage.
- No per-sender overrides, no ledger, no records extraction — later
  milestones. The extraction schema deliberately includes `dates` and
  `people` so the ledger/records milestones can extend rather than replace
  the cache.

## Verification

- Unit: extraction stateKey logic; window selection; link-token
  parser/validator (valid, unknown-ID, >20% rejection); compose-input
  builder caps; defensive JSON parsing (clean, fenced, garbage → retry).
  All pure functions, no provider calls in tests (mock `AiProviderClient`).
- Integration-ish: briefManager trigger gating (feed-only sync → no calls;
  changed focus thread → extraction + compose) with a stub provider.
- `npx tsc --noEmit` clean; full suite green.
- Manual: fresh-run flow, no-key fallback, provider-failure fallback
  (revoke key), link navigation, staleness chip, Ollama smoke test with a
  local model to prove provider-agnosticism.

## Out of Scope

- Morning-notification delivery of the Brief
- Cross-account unified memo (v1 is per active account)
- Sensitive-content redaction before model calls (revisit with Records)
- Brief history/archive of past memos
- Claude classification of ambiguous triage mail (separate milestone-2
  remainder)
